# ダッシュボード（プロジェクト俯瞰ビュー）要件まとめ

## 1. 背景・目的

現状のフロントは「サイドバーで 1 プロジェクトを選択 → 個別の PB 図を見る」構成で、
**一度に確認できるのは個別プロジェクトの状況のみ**。

マネージャー層からは「複数プロジェクトの状況を一覧で俯瞰したい」という要望がある。
そのため、全プロジェクトの進捗サマリを 1 画面で確認できる **ダッシュボード** を新設する。

---

## 2. 画面構成・ナビゲーション

### 2.1 サイドバーへのメニュー追加

- 左上ロゴ（`TestStat Studio`）の **下**、`プロジェクト作成` ボタンの **上** に
  「**ダッシュボード**」メニュー項目を追加する。
- ダッシュボード表示中はメニューを選択状態（`selected`）でハイライトする。
- 既存の「プロジェクト一覧（並び替え可能）」「アーカイブ済み」「設定」はそのまま維持。

```
┌─────────────────────────┐
│  TestStat Studio        │  ← ロゴ（sidebar-header）
├─────────────────────────┤
│  ▦ ダッシュボード        │  ← ★新規追加（ロゴ下・作成ボタン上）
│  ┌───────────────┬──┐   │
│  │ + プロジェクト作成 │ ⟳ │   │
│  └───────────────┴──┘   │
│  ・プロジェクトA          │
│  ・プロジェクトB          │
│  ...                     │
│  ▸ アーカイブ済み (n)     │
├─────────────────────────┤
│  ⚙ 設定                 │
└─────────────────────────┘
```

### 2.2 ルーティング

| URL                      | 表示内容                                     |
| ------------------------ | -------------------------------------------- |
| `/tstat/`                | **ダッシュボード（デフォルト）**             |
| `/tstat/<testing id>`    | 個別プロジェクトの PB 図ページ               |

- `/tstat/` をデフォルトでダッシュボードにする。
- ダッシュボードでプロジェクト行をクリック → `/tstat/<testing id>` の個別 PB 図ページへ遷移。
- 個別ページからサイドバーの「ダッシュボード」メニュー → `/tstat/` に戻る。

> **既存実装との関係（実装時の論点）**
> 現状 `/tstat/<id>` は共有リンク専用で、起動時に id を読み取った後すぐ
> `normalizeUrlToBase()` でアドレスバーを `/tstat/` に正規化している
> （`utils/shareLink.ts`）。本要件では `/tstat/` 自体をダッシュボードの URL とするため、
> 「正規化先 = ダッシュボード」「`/tstat/<id>` = 個別ページ（正規化しない or 維持）」へ
> ルーティング方針を見直す必要がある。実装方式（履歴管理 `pushState`/`popstate` の踏襲か、
> ルーターライブラリ導入か）は別途検討。

## 3. ダッシュボード表示項目

### 3.1 対象プロジェクト

- **アーカイブ済み以外**の全プロジェクトを表示する（`archived === false`）。
- 並び順はサイドバーと同じ `display_order` 昇順を踏襲する。

### 3.2 一覧カラム

各プロジェクトを 1 行（またはカード）で表示し、以下を簡潔に確認できる。

| 項目              | 内容                              | データソース（`ProjectItem`）           |
| ----------------- | --------------------------------- | --------------------------------------- |
| Testing ID        | チケットID                        | `testing_id`                            |
| プロジェクト名    | 表示名                            | `name`                                  |
| ステータス        | 進行中 / 未開始 / 完了            | `has_actuals` / `actual_all_completed`（下記ロジック） |
| 予定期間          | `開始 ~ 終了`（未設定は `-`）     | `planned_start_date` / `planned_end_date` |
| 完了率（対全体）  | 全テストケースに対する完了率      | `actual_completed_rate`                 |
| 完了率（対計画）  | 計画値に対する完了率              | `actual_vs_plan_rate`（`null` は `-`）  |

### 3.3 ステータス判定ロジック（既存踏襲）

`ProjectOverview` / `Sidebar` と同じ判定を流用する。

| 条件                                | ステータス | 色（既存 CSS クラス）       |
| ----------------------------------- | ---------- | --------------------------- |
| `has_actuals === false`             | 未開始     | `not-started`（グレー）     |
| `actual_all_completed === true`     | 完了       | `completed`（グリーン）     |
| 上記以外                            | 進行中     | `active`（ブルー）          |

### 3.4 完了率の表記

- パーセント小数 1 桁（例 `72.5%`）。既存 `formatRate` を踏襲。
- `actual_vs_plan_rate` が `null`（計画未設定）の場合は `-` 表示。
- 視覚補助として、完了率に簡易バー／ドーナツを添えてもよい（モックでは横バーを採用）。

---

## 4. インタラクション

- **行クリック**: 該当プロジェクトの個別 PB 図ページ（`/tstat/<testing id>`）へ遷移。
- 行 hover で背景ハイライト（既存 `project-row:hover` と同テイスト）。
- （任意・将来）ステータス・名称によるフィルタ、カラムソート。本フェーズでは対象外。

---

## 5. デザイン方針

現状フロント（`src/index.css` / `src/App.css`）のトークンをそのまま使用する。

| トークン        | 値        | 用途                     |
| --------------- | --------- | ------------------------ |
| `--bg`          | `#f5f6f8` | 背景                     |
| `--panel`       | `#ffffff` | カード／テーブル背景     |
| `--border`      | `#e3e6ea` | 枠線                     |
| `--text`        | `#1f2933` | 本文                     |
| `--muted`       | `#7b8794` | 補足テキスト・ヘッダ     |
| `--accent`      | `#274c77` | アクセント（紺）         |
| `--accent-soft` | `#eaf2fb` | 選択・hover の淡い背景   |

- フォント: `"Segoe UI", "Hiragino Kaku Gothic ProN", "Meiryo", sans-serif`
- ステータスのピル、テーブルのヘッダ（`#f6f8fb` 背景・muted 文字）など既存スタイルを流用。
- レスポンシブ: 760px 以下ではサイドバーを上部に畳む既存挙動を踏襲。

---

## 6. データ取得

- ダッシュボードは既存 `GET /api/v1/projects`（`fetchProjects`）の結果をそのまま利用可能。
  `ProjectItem` に必要項目（`actual_completed_rate` / `actual_vs_plan_rate` /
  `planned_*` / `has_actuals` / `actual_all_completed`）が既に含まれているため、
  **新規 API は不要**の見込み。
- サイドバーと同じデータソースを共有し、再読込（`onRefresh`）も共通化する。

---

## 7. スコープ外（本フェーズで作らないもの）

- カラムソート／フィルタ／検索（将来拡張）。
- 集計グラフ（プロジェクト横断の合算チャート等）。
- 不具合（バグ）件数の俯瞰列。`bug_count_source` 連携とあわせて将来検討。
- 権限・ロール別表示。
